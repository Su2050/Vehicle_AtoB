## 二、方法论：实验驱动交付

严格遵循 **单因素实验法**：

1. **建立 Baseline** — 复现当前结果，确认入口命令与通过率
2. **列出因素清单** — 标注可控/不可控、验证成本、优先级
3. **单因素改变** — 每轮只改 1 个变量，禁止顺手重构
4. **假设-验证闭环** — 写清假设、最小 diff、验证命令、通过/失败标准
5. **实验日志** — 记录变量、改动、观察、结论、下一步

### 阶段 3: 启发式函数深度诊断（Exp-4a ~ Exp-4c）

> 这是整个优化过程中**最关键的突破口**。

#### Exp-4a: A\* 搜索效率根因分析 🔍

深入分析发现 **Dijkstra 网格膨胀半径严重不匹配**:

```
DijkstraGrid 膨胀半径: 0.7m（硬编码）
车辆实际碰撞半径:      ~0.1m
比例:                  7:1 ❌
```

**后果链条**:
1. 障碍物周围 0.7m 范围内所有网格单元被标记为 `obs_map = True`
2. `get_heuristic()` 对这些单元返回惩罚值 `1000.0`（相当于禁区）
3. A\* 无法探索障碍物附近的绕行路径
4. 搜索被困在远离障碍物的外围区域，无法找到穿越间隙的路径

#### Exp-4b: 修复 Dijkstra 膨胀半径 ✅

| 项目 | 内容 |
|------|------|
| **变量** | Dijkstra 网格的障碍物膨胀半径 |
| **假设** | 将膨胀从 0.7m 降至 0.25m，让 A\* 能探索障碍物附近路径 |
| **改动文件** | `heuristic.py`: 构造函数接受 `inflate_radius` 参数<br>`planner_obs_v2.py`/`planner_obs.py`: 传入 `inflate_radius=0.25` |
| **结果** | A\* 搜索空间打开，但仍在 S02 上超时 |

#### Exp-4c: 改进膨胀区域启发式 ✅

| 项目 | 内容 |
|------|------|
| **变量** | 膨胀区域内的启发式返回值 |
| **假设** | 不应返回 1000.0 禁区值，应返回近似欧氏距离 |
| **改动** | 预计算 `inflated_dist_map`，膨胀区返回欧氏估计而非禁区值 |
| **结果** | 启发式更合理，A\* 搜索方向改善 |

**阶段 3 结论**: 膨胀半径修复打开了搜索空间，启发式改进让搜索方向更合理。但 A\* 的**细粒度运动基元**（每步仅移动 0.02m 横向）仍然是效率瓶颈。

#### Exp-5b: 航向拼接回溯机制 ✅

| 项目 | 内容 |
|------|------|
| **变量** | `_stitch_waypoints` 最后一段目标航向对齐 |

## 四、关键技术发现

### 4.1 Dijkstra 膨胀半径是搜索效率的"隐形杀手"

```
膨胀 0.7m（原始）→ 障碍物周围大片区域被标为禁区
                  → 启发式返回 1000.0
                  → A* 完全无法探索绕行路径
                  → 搜索被困在外围 → TIMEOUT

膨胀 0.25m（修复）→ 障碍物周围仅保留安全裕量
                   → 启发式返回合理欧氏估计
                   → A* 能探索绕行间隙
                   → 搜索成功或快速转入回退
```

**教训**: 膨胀半径不是越大越安全，过大的膨胀会让搜索算法"失明"，反而更危险。应与车辆实际尺寸匹配。

### 4.2 2D 骨架回退是复杂障碍的终极武器

A\* 基于细粒度运动基元（0.02m 步长），在需要大范围绕行时步数爆炸。而 2D 骨架：

1. **Dijkstra 距离场追踪** — 粗粒度 2D 路径，天然考虑障碍物
2. **路点简化** — 将数千个网格点压缩为 5~15 个关键路点
3. **RS 分段拼接** — 用 Reeds-Shepp 曲线连接路点，保证运动学可行

相比 A\* 的数万次节点扩展，2D 骨架通常在 1~3 秒内完成。

### 4.3 时间预算分层是鲁棒性的关键

```
❌ 错误做法: 所有阶段共享 30s 总预算
   → Stage-1 A* 独占 27s → L1.8 只剩 3s → TIMEOUT

✅ 正确做法: 分阶段独立预算
   → Stage-1: 8s（快速尝试，失败则放弃）
   → L1.8: 15s（2D 骨架主力）
   → Stage-2: 10s（精细调整）
   → Level-3: 剩余时间兜底
```

### 4.4 条件门控的"一字之差"

```python
# ❌ 错误: 单障碍物场景无法触发 2D 骨架
run_2d_skeleton = obstacles and not prefix_too_many_shifts and len(obstacles) > 1

# ✅ 修复: 只要有障碍物就应该尝试
run_2d_skeleton = bool(obstacles) and not prefix_too_many_shifts
```

S02（单个方块）从全军覆没到全部通过，仅仅因为删掉了 `and len(obstacles) > 1`。

### 4.5 `_stitch_waypoints` 的航向困境

当 `MIN_TURN_RADIUS=2.1m` 且剩余路径只有几米时，RS 曲线无法在短距离内大幅改变航向。解决方案是**回溯拼接**：

```
路点序列: A → B → C → D → Goal

正常拼接: A→B ✅, B→C ✅, C→D ✅, D→Goal ❌ (航向偏差太大)

回溯拼接: 尝试 C→Goal ✅ (从更远处直连，有足够距离调头)
最终路径: A→B→C→Goal ✅
```

## 七、经验总结

1. **实验驱动 > 直觉修复** — 单因素实验法虽然看起来慢，但每一步都有明确的排除/确认结论，避免了"改了一堆但不知道哪个有效"的困境
2. **启发式质量决定搜索成败** — A\* 的效率 90% 取决于启发式函数的引导质量。膨胀半径错误导致的禁区标记，比任何 expand_limit 提升都更致命
3. **分层回退比单一算法更鲁棒** — L1(RS) → L1.5(Milestone) → L1.8(2D骨架) → L2(A\*) → L3(兜底) 的瀑布式架构，确保不同复杂度的场景都能找到对应层级处理
4. **时间预算必须分层隔离** — 共享预算导致"大户吃小户"，各阶段独立预算才能保证每个层级都有机会执行
5. **诊断脚本是定位问题的利器** — 单独的 `_diag_l18.py` 在隔离环境中快速验证了 `plan_2d_fallback` 的能力，精准定位了 spacing 和 deadline 两个配置问题
