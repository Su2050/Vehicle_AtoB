# 叉车路径规划算法超时与“揉搓”问题修复总结

## 1. 背景与现象

在叉车避障路径规划（基于 A* 搜索、运动基元与 Reeds-Shepp 曲线的混合多阶段规划器）的测试中，我们遇到了两个严重影响成功率和表现质量的核心问题：

1. **高频超时卡死 (Timeout / Impossible)**：在带有障碍物的复杂场景（如用例 `#055`）中，规划器经常展开数万甚至数十万个节点后依然找不到路径，最终触发超时被强行终止。
2. **“愚蠢”的反复揉搓动作 (Zigzagging)**：在部分生成的路径中，叉车会在极小的空间内频繁前进、倒退（F-R-F-R），产生了大量无效的冗余动作，看起来非常不智能。

---

## 2. 根因深度剖析 (Root Cause Analysis)

通过引入分阶段耗时统计（Phase-Level Statistics）并对卡死用例进行逐步 Debug，我们发现了隐藏在底层的几个致命冲突：

### 2.1 Stage-1.5 的“死亡夹击”
原本设计 `Stage-1.5` 的目的是为了消化 `Stage-1` 绕过障碍物后遗留的巨大 Y 轴偏差。
- **冲突点**：`Stage-1.5` 施加了极其严格的 X 轴空间限制（`_goal_xmax` 被卡死在非常靠近车身的位置）。当叉车为了消除 Y 偏差而尝试倒车转向时，由于阿克曼几何特性，车头会向外大幅甩动，直接撞上走廊的虚拟侧墙。
- **结果**：向后退会撞墙，向前开又受限于 X 轴上限。A* 算法被物理限制和代码约束卡死在了一个极小的空间内，只能在原地疯狂进行前后微调扩展节点，瞬间耗尽计算资源导致 Timeout。

### 2.2 启发式陷阱：“海妖之歌” `h_rs`
A* 算法依赖启发式函数来指引搜索方向。
- **冲突点**：在 Stage-2 中，我们使用了 Reeds-Shepp 曲线距离 (`h_rs`) 作为启发式。**但 `h_rs` 是纯运动学的，完全无视障碍物！**
- **结果**：`h_rs` 像海妖一样，疯狂诱导 A* “抄近道”直接穿过障碍物去向目标。当 A* 按照指引扩展节点时，又被底层的物理碰撞检测 (`check_collision`) 无情驳回。这种“高层指引”与“底层物理”的严重撕裂，造就了深不见底的局部最优陷阱（Local Minimum）。

### 2.3 Dijkstra 膨胀半径过小
- **冲突点**：用于提供网格引导的 `DijkstraGrid`，其障碍物物理膨胀半径被设定为 `0.5m`（等于叉车自身的静态半径）。
- **结果**：在直线行驶时没问题，但在转弯时，叉车车体会扫过更大的区域。偏小的膨胀半径导致 Dijkstra 认为某些贴着障碍物的路径是“安全”的，误导 A* 走入物理上会发生碰撞的死胡同。

### 2.4 Stage-2 缺失避障能力的“假成功”
- **冲突点**：在基线代码中，最终对齐阶段 `Stage-2` 调用 `plan_path` 时，硬编码传入了 `obstacles=None`。
- **结果**：这意味着最后阶段生成的 RS 曲线尾巴如果扫到了障碍物，系统根本不知道。导致大量测试用例在日志中显示 `Success: True`，但在实际物理可视化中却是穿模撞墙的。

---

## 3. 终极修复方案 (The 4-Point Ultimate Fix)

为了保证“成功的继续成功，失败的也能成功”，我们实施了以下组合拳，彻底重构了对齐逻辑：

### 🛠️ 1. 废除残缺的 Stage-1.5，赋予 Stage-2 真正的避障能力
- **删除**了原本受限且不带避障的 `Stage-1.5` A* 搜索。
- **引入**了一段轻量级的 **贪心 K-turn (Greedy K-turn)** 作为阶段间的过渡。利用纯几何规则强制消除 Stage-1 遗留的巨大 Y 偏差。
- **修复**了 `Stage-2` 的漏洞，现在强制传入真实的障碍物数据 (`obstacles=fast_obstacles`)，确保最终生成的每一段曲线都 100% 物理合法，消除了“穿模假成功”。

### 🛠️ 2. 解除空间紧箍咒，解锁前向揉库能力
- 将最终阶段的 `_goal_xmax` 彻底放开（提升至 `8.0`），并修复了 `dyn_xmax` 导致的空间截断 Bug。
- 允许叉车在需要调整姿态时，有充足的**前向空间**开出去，从而顺畅地完成 K-turn，再也不用在原地“戴着镣铐跳舞”。

### 🛠️ 3. 修复启发式陷阱 (Heuristic Traps)
- **增加 Dijkstra 膨胀**：将物理膨胀半径从 `0.5m` 提升至 `0.7m`，柔性膨胀提升至 `1.2m`。让网格引导更加保守，提前避开转弯时可能碰撞的危险区域。
- **限制 `h_rs` 的触发时机**：将盲目无视障碍物的 `h_rs` 触发距离从 `6.0m` 大幅压缩到了 `2.0m`。只有在极其靠近目标、确信视距内没有任何阻挡时，才允许 `h_rs` 介入引导，防止其在远距离把搜索带进沟里。

### 🛠️ 4. 智能“局部抗揉搓”代价 (Smart Anti-Zigzag Penalty)
为了解决叉车动作“蠢”的问题，同时不误伤合法的操作：
- 我们没有采用全局的高昂换挡惩罚（那会导致叉车不敢换挡而卡死）。
- **实现方案**：引入了历史回溯窗口。在 A* 计算代价时，向上追溯最近的 6 步轨迹，统计换挡次数。
- **逻辑**：一个正常且必须的 K-turn（进-退-进 或 退-进-退）正好包含 2 次换挡，系统**允许**这样做。但如果在这极短的窗口内检测到 **第 3 次（含以上）换挡**（例如 F-R-F-R，典型的无意义原地揉搓），系统将施加 `+15.0` 的严厉惩罚。
- **效果**：完美封杀了愚蠢的连续锯齿动作，同时保留了叉车在狭窄空间内正常揉库的灵活性。

---

## 4. 结论

通过引入回归门测试框架，并利用分层日志精确定位了底层机制冲突。这套重构方案不仅大幅提升了带障碍物场景下的路径规划成功率，保证了物理可行性，还极大地减少了无意义的节点展开，优化了规划速度和路径的美观度。系统现在的表现更加鲁棒且“智能”。
