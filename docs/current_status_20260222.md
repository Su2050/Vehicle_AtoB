### 2.2 分阶段耗时特征（失败用例）

典型失败用例的阶段统计格式：`(P0:0ms, S1:0ms, S1.5:Xms, S2:0ms)`

- **P0 (Phase-0)**: 0ms — 掉头预处理未触发或极快完成
- **S1 (Stage-1)**: 0ms — 预定位/避障阶段被跳过（起点已在预接近区或无障碍需穿越）
- **S1.5 (Stage-1.5)**: 几十到几百 ms — 贪心 K-turn 做 Y 轴对齐，耗时很短
- **S2 (Stage-2)**: 0ms — 最终 RS 对准阶段未成功，直接进入 fallback 单阶段 A*

**结论**：失败主要发生在 **fallback 单阶段 A*** 中。Stage-1.5 完成后，Stage-2 的 `plan_path` 未能找到可行路径，于是回退到全局 A*，该 fallback 要么达到 `_expand_limit`（如 150000）后返回 No path found，要么在 45s 内未完成而超时。

## 5. 当前仍存在的问题

### 5.1 成功率极低

在 quick 档位、7 场景、39 用例下，成功率仅约 12.8%，远低于预期。

### 5.2 Fallback A* 成为瓶颈

- Stage-2 的 `plan_path`（带 RS 扩展、带障碍物）经常失败
- 失败后进入 fallback 单阶段 A*，该路径搜索空间大、约束多，容易：
  - 达到 `_expand_limit`（如 150000）仍无解
  - 或单用例 45s 内无法完成而超时

### 5.3 可能的根因方向（待验证）

1. **Stage-2 的 RS 扩展与障碍物冲突**：RS 曲线在障碍物附近被大量驳回，导致 Stage-2 难以命中目标，频繁回退到 fallback
2. **启发式与障碍物不匹配**：即便限制了 `h_rs` 触发距离，在部分姿态下仍可能把搜索导向不可行区域
3. **`check_collision` 边界约束**：此前分析发现，在 `nx > 2.05` 时，若 `no_corridor` 相关逻辑有遗漏，可能缺少全局 Y 边界（如 y ∈ [-3.5, 3.5]），导致 A* 向无效区域扩展
4. **DijkstraGrid 越界启发式**：节点在网格外时，若启发式返回 0 或过小，会误导 A* 向“地图外”探索

## 6. 建议的后续动作

1. **确认并修复 `check_collision` 的全局物理边界**：无论 `no_corridor` 如何，都应强制 `ny ∈ [-3.5, 3.5]`、`nx ∈ [-0.5, 9.0]`
2. **确认 DijkstraGrid 越界惩罚**：网格外节点应返回足够大的启发式，避免“免费”探索
3. **针对典型失败用例做单点调试**：如 `(3.5, 0.5, 0°)`、`(3.5, 1.5, 0°)`，用 `test_specific_case.py` 复现并逐步排查
4. **评估 Stage-2 的 `_expand_limit` 与超时策略**：在保证不拖垮整体测试的前提下，是否需要对 Stage-2 或 fallback 做差异化限制或更早终止
