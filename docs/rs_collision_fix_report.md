# RS 轨迹碰撞假阳性 (False Positives) 修复报告

## 问题背景
在之前的压力测试 (Stress Test) 中，发现规划器输出了多达 40 个“假阳性”碰撞用例 (`TRAJ_COLLISION`)。即：**规划器内部认为路径是安全的并返回了成功，但通过真实的物理模拟回放时，车辆实际上发生了碰撞**（例如撞墙或穿过安全走廊）。

这些错误绝大多数发生在终点前的最后一段调整，或复杂的绕行过程中。

---

## 🔍 深度分析：问题的根源到底是什么？

经过多因素与单因素实验交叉验证，发现导致此问题的根源是由**两个深层 Bug 叠加**造成的：

### 1. 根源一：多路径 RS 采样 (`rs_sample_path_multi`) 坐标系反转错误
在处理避障多路径生成时，数学模型为了适配标准的 Reeds-Shepp (RS) 曲线，对目标的局部坐标 `lx`, `ly` 进行了硬编码的反转：
```python
x_rs_goal = -lx
y_rs_goal = -ly
```
* **现象**：虽然在纯数学的相对空间里曲线两端连上了，但映射回真实世界坐标系时，生成的轨迹和车辆的实际物理朝向产生了畸变与错位。这也是为什么很多碰撞总是发生在终点附近最后一段 `rs_traj` 里的原因。
* **解决**：移除了错误的负号硬编码，严格按照前向局部坐标系 (`forward=+x`) 重新对齐了 `rs_sample_path_multi`，让数学轨迹正确地回归到物理现实世界。

### 2. 根源二：V2 避障规划器 (`planner_obs_v2.py`) 中的“偷吃步”穿模
在遇到复杂障碍物时，V2 规划器会生成 Milestone（绕行路点）并用多段 RS 曲线拼接。为了提高找路成功率，代码中对中间的 RS 拼接段**错误地使用了放宽的碰撞检测**：
```python
# 旧代码中的错误：关闭了走廊和物理边界
coll_relaxed = lambda nx, ny, nth: check_collision(
    nx, ny, nth, no_corridor=True, obstacles=fast_obstacles)
```
* **现象**：它设置了 `no_corridor=True`（关闭走廊/边界碰撞检测），导致车辆在绕行路点时，规划器**默许了轨迹穿透了安全走廊甚至物理墙壁**。规划器以为自己找到了一条绝妙的捷径并返回成功，但实际上这是一条物理上会撞墙的“穿模”路径。
* **解决**：去除了这种不安全的放松机制，将其改为了强制使用上下文传递进来的严格 `collision_fn`（包含安全走廊与全物理边界验证），绝不允许在任何拼缝处关闭物理碰撞检测。
