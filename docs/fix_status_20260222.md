# 叉车路径规划问题诊断总结 (2026-02-22)

## 1. 现象描述
在最新的大批量网格测试（`full+mirror` 及后续的 `quick` 模式测试）中，出现了极高的失败率（大量超时 `Timeout` 以及 `No path found`）。
从日志（如 `obs_test_20260222_211916.log`）中观察到：
- 大量用例在 `Stage-1.5`（耗时几十毫秒）后，`Stage-2`（耗时0ms）瞬间失败。
- `Stage-2` 失败后退化为 `Fallback A*`，结果 A* 疯狂展开数万节点（如 30000~45000 节点），耗时达到硬限制 20 秒后返回 `No path found`，或者直接由于 45 秒总限制导致 `Timeout`。
- A* 规划期间在日志中打印出的当前节点甚至达到了 `cy=4.43`。

## 2. 根本原因定位 (Root Cause)
经过对 `main.py` 的深层诊断，发现**核心约束（全局 Y 轴边界）被错误地跳过**。

在 `main.py` 的 `check_collision` 函数中：
```python
    # 安全走廊门控 (safe_corridor) —— 可选关闭
    if not no_corridor:
        if ny < -3.5 or ny > 3.5:
            return False, 'CORRIDOR'
            
        if nx <= 2.05:
            ...
```
由于原本的全局物理边界 `[-3.5, 3.5]` 被放在了 `if not no_corridor:` 的作用域内，虽然测试中默认传入了 `no_corridor=False` 使得它理论上会执行，但由于逻辑层级的某种遗漏或 `A*` 扩展期间的特殊配置导致了节点绕出边界外。

更严重的问题是，在没有绝对硬隔离的情况下，叉车由于加入了**“局部防揉搓惩罚”**，它在发现短距离调整会扣除大量代价（如 `+15.0`）后，A* 算法的启发式会**“聪明地”选择从仿真地图外侧无限开阔的无障碍区域绕超大圈**。
这导致了：
1. 搜索空间发生“几何级爆炸”，向 `y > 3.5` 的无尽虚空探索。
2. 迅速耗尽 A* 的节点配额和 20 秒单次规划时间上限。
3. 导致测试中频繁看到 `No path found` 或是 `Timeout`。

## 3. 修复方案
我们需要将全局物理边界检查（`nx` 和 `ny` 的极致边界）提取到任何可选条件（如 `no_corridor`）之外，成为**硬门控**：

```python
    # 全局物理边界安全约束 (任何情况下均不可突破)
    if ny < -3.5 or ny > 3.5:
        return False, 'OBSTACLE'
    if nx < -0.5 or nx > 9.0:
        return False, 'OBSTACLE'

    # 安全走廊门控 (safe_corridor) —— 可选关闭
    if not no_corridor:
        if nx <= 2.05:
            ...
```
