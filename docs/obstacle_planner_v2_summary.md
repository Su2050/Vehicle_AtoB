# 障碍物规划器 v2 — 重构与迭代总结

## 1. 背景与目标

原有障碍物规划器 (`planner_obs.py`) 在简单障碍场景下可用，但面对长墙绕行、U 型陷阱脱困、L 形障碍迂回等复杂布局时，由于 `astar_core.py` 的细粒度状态离散化（0.02m XY、1.15° 航向）导致搜索空间爆炸，频繁超时（>20s）。

本次重构参考了 `obc_alg/` 中 **Ring-Band RS Planner** 的核心思路，在 **不修改任何现有文件** 的前提下，通过新建模块和回调注入的方式实现了增强版规划器 `planner_obs_v2.py`。

## 2. 架构设计原则

- **零侵入**：所有增强均通过新模块实现，利用 `astar_core.py` 的回调架构（`collision_fn`、`heuristic_fn`、`rs_expand_fn`）注入逻辑
- **分层兜底**：从快到慢逐级尝试，尽早得到解
- **借鉴但不照搬**：从 `obc_alg` 提取可迁移的思想，适配当前项目的单圆碰撞模型和回调架构

## 3. 新建文件

| 文件 | 用途 |
|------|------|
| `scripts/planner_obs_v2.py` | 增强版障碍物规划器主模块（~590行） |
| `scripts/fallback_2d.py` | 2D 骨架路径 + RS 拼接兜底模块（~400行） |
| `scripts/tests/test_obs_v2.py` | 综合测试套件（12 场景 × 37 用例） |

## 4. 从 obc_alg 借鉴的核心技术

### 4.1 RS 多候选扩展

**obc_alg 做法**：枚举全部 48 种 Reeds-Shepp 路径类型，按长度排序逐一碰撞检测。

**v2 适配**：`_make_rs_expand_fn_multi()` 调用 `rs.rs_sample_path_multi()` 生成多条 RS 候选路径（5-15 条），遇到碰撞的跳过而非直接放弃。解决了单一最短 RS 路径被障碍物阻挡导致扩展失败的问题。

### 4.2 进展感知启发式

**obc_alg 做法**：`D_goal(candidate) ≤ D_goal(current) + allow_uphill` 约束，防止搜索远离目标。

**v2 适配**：`_make_heuristic_fn_v2()` 基于 Dijkstra 距离场实现软进展约束。当节点的 Dijkstra 距离显著大于当前最优时，施加额外代价惩罚（而非硬性剪枝），配合可调的 `h_weight`（2.0-4.0）引导搜索方向。

### 4.3 多阶段恢复梯子

**obc_alg 做法**：多级参数配置，从保守到激进递增。

**v2 适配**：`RECOVERY_STAGES` 定义三级恢复策略：

| 阶段 | expand_limit | RS 半径 | allow_uphill | h_weight |
|------|-------------|---------|-------------|----------|
| fast | 8,000 | 4.0m | 2.0 | 4.0 |
| robust | 25,000 | 6.0m | 6.0 | 3.0 |
| aggressive | 80,000 | 10.0m | 100.0 | 2.0 |

### 4.4 里程碑 RS 绕行（Level-1.5）

**obc_alg 做法**：在障碍物周围采样里程碑点，构建里程碑图搜索。

**v2 适配**：`_try_milestone_rs_bypass()` 在障碍物四周生成密集候选路点网格，尝试 2 段（起点→路点→目标）和 3 段（起点→绕行点→回归点→目标）RS 路径拼接。中间段使用放松碰撞（`no_corridor=True`），最终段使用严格碰撞。

### 4.5 2D 骨架路径 + RS 拼接（Level-1.8）

**v2 原创**：针对 `astar_core` 细粒度搜索在复杂障碍下的性能瓶颈，实现了完全独立于 A* 的兜底方案：

1. **梯度路径追踪**：沿 DijkstraGrid 距离场梯度从起点回溯到目标，得到粗 2D 骨架
2. **代价惩罚 Dijkstra**：独立的 2D 搜索器，对障碍物周围 `MIN_TURN_RADIUS × 1.2` 范围施加通行代价惩罚，自动避开 RS 曲线无法通过的窄缝隙，迫使路径走宽侧
3. **RS 分段拼接**：将骨架简化为路点后，逐段用 Reeds-Shepp 曲线连接，中间段支持宽范围航向搜索（±180°），最后一段强制精确对准目标航向
4. **多起点策略**：同时从 K-turn 后位置和原始起点两个位置尝试，避免 K-turn 预处理将车辆移至更差位置

### 4.6 A* 后验检查

**问题**：`astar_core.plan_path` 使用宽目标区域（`_goal_xmin=0.5, _goal_xmax=8.0`），当 A* 原语进入该区域但未执行 RS 扩展时，可能产生假阳性（声明成功但实际未到达托盘）。

**修复**：在 Level-2 A* 返回 `ok=True` 但无 RS 轨迹时，检查 A* 找到的 `goal_pos` 是否在窄目标区域（`x ≤ 2.25, |y| ≤ 0.18, |θ| ≤ 5°`）。如不在，尝试从该位置向真实托盘做 RS 连接验证，失败则判定当前阶段无效并继续下一阶段。

## 5. 完整规划层级

```
Level-1    : Pure RS 直连（无障碍阻挡时 <1ms）
Level-1.5  : 里程碑 RS 绕行（简单障碍 ~200ms）
  ↓ Phase-0 转向 + K-turn 预处理
Level-1.8  : 2D 骨架 + RS 拼接（复杂障碍 0.8-3s）
Level-2    : 多阶段 A*（fast → robust → aggressive）
Level-3    : 从原始起点的 A* 全局兜底
```

### 6.2 场景设计思路

**典型场景（S01-S04, S07, S08, S10）**：覆盖常见仓库布局，验证基本功能正确性。

**"容易翻车"场景（S03, S05, S09, S11）**：

- **S03 长墙**：Dijkstra 最短路径经过底部窄缝（<2m），RS 曲线无法通过（最小转弯半径 2.1m）。需要代价惩罚机制将路径推向顶部宽侧。
- **S05 U 型陷阱**：车辆被三面包围，必须先向开口方向（+x）退出，再大幅绕行到目标。考验脱困能力。
- **S09 死胡同**：面朝死端时必须倒车退出。考验倒车规划能力。
- **S11 L 形障碍**：竖墙 + 横顶形成的 L 形，只有一个缺口可通过。考验方向变换和精确穿越。

**边界场景（S06, S08, S12）**：

- **S06 不可达**：全高墙完全封锁目标，规划器必须正确返回失败。曾因 A* 宽目标区域产生假阳性，通过后验检查修复。
- **S08 起点=目标**：退化情形，应瞬间返回。
- **S12 强制兜底**：人为压低 `_expand_limit` 到极低值，确认 A* 失败后兜底机制（里程碑 RS 或 2D 骨架）仍能出解。

### 6.3 随机障碍生成

`_generate_random_obstacles(seed)` 使用确定性种子生成最多 8 个障碍物，同时在 `y=0` 附近刻出 1.8m 宽的保障走廊，确保存在可行路径。三个不同 seed（42、123、2026）测试随机分布下的鲁棒性。

### 6.4 验证机制

每个用例运行后执行以下检查：

1. **成功/失败匹配**：比较 `ok` 返回值与预期（`expect_success`）
2. **RS 轨迹碰撞检测**：对返回的 RS 轨迹逐点做严格碰撞检测（含 corridor 约束）
3. **超时检测**：通过 `signal.SIGALRM` 设置每个用例的超时上限（默认 45s）
4. **报告输出**：每个用例记录耗时、命中层级（Level）、详细信息，汇总为全局通过率

### 6.5 运行方式

```bash
# v1 vs v2 对比

```

## 8. 迭代过程中的关键调试问题

### 8.1 corridor 约束导致 RS 段被拒

`collision.py` 中的 "safe corridor" 在 `x ≤ 2.05` 时限制横向移动。2D 骨架和里程碑绕行的中间 RS 段可能经过该区域被误判碰撞。

**解决**：中间段使用 `no_corridor=True` 的放松碰撞函数。

### 8.2 DijkstraGrid 膨胀半径硬编码

`heuristic.py` 的 `build_map()` 中障碍物膨胀半径固定为 0.7m，不受构造参数控制。导致无法通过增大膨胀来排除窄通道。

**解决**：在 `fallback_2d.py` 中实现独立的 `_dijkstra_2d_path()` 搜索器，支持可配置的硬膨胀 + 软代价惩罚，自动将路径推离窄缝隙。

### 8.3 K-turn 预处理改变起点

K-turn 预处理可能将车辆移到比原始位置更不利的位置（如更靠近障碍物），导致后续 2D 骨架兜底从新位置出发时失败。

**解决**：Level-1.8 同时从 K-turn 后位置和原始起点两个位置尝试 2D 骨架。

### 8.4 RS 最短路径航向偏差

`_try_connect_with_heading_variants()` 优化最短路径长度，但可能选择航向偏差极大的变体（如 -57° vs 目标 0°），导致终点验证失败。

**解决**：最后一段 RS 连接强制使用精确目标航向（仅允许 ±0.08 rad 微小偏差）。
